LOG ANALYSIS TASK (CRITICAL):

Log lines are typically key=value pairs separated by spaces.

--------------------------------------------------
STEP 1 — PARSE KEY=VALUE TOKENS (PRIMARY)
--------------------------------------------------

- Parse each line by splitting on whitespace
- For each token containing '=':
    split only on the first '='
    store key and value in a dict

Recommended pattern:

    parts = line.strip().split()
    data = {}
    for p in parts:
        if '=' in p:
            k, v = p.split('=', 1)
            data[k] = v

--------------------------------------------------
STEP 2 — EXTRACT CORE FIELDS
--------------------------------------------------

- Extract method, endpoint, status, and latency from parsed fields
- Prefer direct keys when present
- Strip whitespace from extracted values

Preferred semantic keys:

- method:
    method, http_method, verb

- endpoint:
    endpoint, path, route, url, uri

- status:
    status, status_code, code, response_code

- latency:
    latency_ms, latency, response_time_ms, response_time, duration_ms, duration

--------------------------------------------------
STEP 3 — SAFE PARSING
--------------------------------------------------

- Never fail because one line is malformed
- If endpoint is missing after normal extraction, try regex fallback from the raw line
- If status cannot be parsed, skip the line or use a safe fallback only if clearly valid
- For latency:
    - do not blindly default missing latency to 0
    - first try to extract a numeric value from the latency-like field
    - if no numeric latency can be found, skip latency contribution for that line

Recommended latency parsing:

    raw_latency = ...
    raw_latency = str(raw_latency).strip()

    if raw_latency:
        m = re.search(r'(\\d+(?:\\.\\d+)?)', raw_latency)
        if m:
            latency = float(m.group(1))
        else:
            latency = None
    else:
        latency = None

--------------------------------------------------
STEP 4 — ENDPOINT FALLBACK
--------------------------------------------------

- If endpoint is missing from parsed fields:
    try extracting a route-like value from the raw line using regex

Example fallback:
    /api/...

- If endpoint is still missing, skip the line

--------------------------------------------------
STEP 5 — GROUPING
--------------------------------------------------

- Group by (method, endpoint)

For each group compute:
- total_requests = total line count in that group
- avg_latency = average of valid latency values only
- error_rate = count of status >= 400 divided by total_requests
- p95_latency = 95th percentile of valid latency values only

- Store all valid latency values for each group
- Do not let missing latency values become artificial zeros

--------------------------------------------------
STEP 6 — PERCENTILE
--------------------------------------------------

- Sort latency values ascending
- Compute p95 from the sorted valid values

Recommended pattern:

    values = sorted(latencies)
    index = int(0.95 * (len(values) - 1))
    p95_latency = values[index]

- If no valid latency values exist for a group:
    use 0 only at final output stage

--------------------------------------------------
STEP 7 — SORTING
--------------------------------------------------

- Rank groups by error_rate descending
- Return the top 5 groups
- If needed, use total_requests descending as a secondary tiebreaker

--------------------------------------------------
STEP 8 — OUTPUT
--------------------------------------------------

- Return a table-like result or structured rows containing:

{
    "method": "...",
    "endpoint": "...",
    "total_requests": int,
    "avg_latency": float,
    "error_rate": float,
    "p95_latency": float
}

- Round numeric metrics appropriately for readability

--------------------------------------------------
IMPORTANT RULES
--------------------------------------------------

- Process the full log, not just a sample
- Do not hardcode a single latency field name
- Do not replace missing latency with 0 before aggregation
- Prefer valid parsed latency values only
- Returning empty results is invalid
